# 029: Functional Verification — Playwright Tests Driven by the Server

**Status:** done

## Goal

The build pipeline validates code quality (unit tests, adversarial review) but never validates that features actually work from a user's perspective. Spec 022 was built and deployed 3 times — all tests pass, Gemini says PASS — but the project selector dropdown never appeared in the dashboard. This spec adds Playwright-based functional testing to the deterministic pipeline so the server can verify that built features are real, not just code that compiles.

## Architecture

### Pipeline Change

```
BEFORE:
Claude builds → pytest (unit) → adversarial review → result

AFTER:
                                    ┌─ Gemini writes Playwright test
Claude builds → pytest (unit) ──────┤
                                    └─ Server writes test to worktree
                                              │
                                    pytest runs ALL (unit + functional)
                                              │
                                    adversarial review (gets Playwright results)
                                              │
                                           result
```

### The Flow

1. **Server sends to Claude**: "Build spec 022" (existing flow, with north star prompt)
2. **Server sends to Gemini** (new step, after Claude finishes building):
   - Mission statement (first sentence of Goal — already extracted by `spec_parser.py`)
   - Unchecked Done When items (what should be testable)
   - Dashboard URL (`http://localhost:8014` inside Docker network)
   - A pytest-playwright template showing the expected pattern
   - "Write a pytest-playwright test that verifies the mission is achieved. Return a single test file."
3. **Server writes Gemini's test** to `tests/functional/test_spec_{number}.py` in the worktree
4. **Server runs pytest** (existing `run_tests()`) — now picks up both unit and functional tests
5. **If Playwright test fails**: Claude gets the failure output via existing triage flow, tries to fix
6. **Adversarial review** receives all test results including Playwright pass/fail counts
7. **Loop**: if adversarial FAILs, Claude fixes, tests re-run (existing loop — Playwright tests included)

### What Gemini Gets (Test-Writing Prompt)

```
You are writing a functional test for a web application using pytest-playwright.

## Mission
{first sentence of spec Goal}

## Features to Verify
{unchecked Done When items}

## Test Target
URL: http://localhost:8014
Framework: pytest with playwright (pytest-playwright package)

## Template
```python
from playwright.sync_api import Page, expect

def test_mission_achieved(page: Page):
    page.goto("http://localhost:8014")
    # Verify the mission is achieved from a user's perspective
    ...
```

Write tests that verify user-observable behavior — what a person would see and click.
Do NOT test internal APIs, database state, or implementation details.
Each test should map to a Done When item where possible.
Return a single Python file.
```

### Infrastructure Requirements

- **pytest-playwright**: pip package, wraps Playwright in standard pytest. Compatible with existing `run_tests()`.
- **Playwright browsers**: `playwright install chromium` — headless Chromium, ~150MB. Installed in Dockerfile.
- **Dashboard must be running** during test: for dashboard specs, the container is live on the Docker network. For other web projects, a test server is started before Playwright runs.
- **Network**: Playwright in the dashboard container can reach `localhost:8014` (itself) or other containers via Docker network names.

### Where Tests Live

```
tests/
  functional/          # Playwright tests (generated by Gemini per-spec)
    conftest.py        # Shared fixtures (base_url, browser config)
    test_spec_022.py   # Generated: project selector works
    test_spec_025.py   # Generated: queue redesign works
    ...
  harness/             # Existing unit tests
  runtime/             # Existing unit tests
  adversarial/         # Existing (Gemini-generated code review tests)
```

### When Functional Tests Run

Not every spec needs Playwright. The server decides based on spec content:
- **Has frontend changes** (`.html`, `.js`, `.css` in the diff) → run functional tests
- **Has API-only changes** (`.py` only, no frontend) → skip Playwright, unit tests suffice
- **Has a running service** (dashboard, your-project) → Playwright can reach it
- **Is a library/CLI** (no web UI) → skip Playwright entirely

This decision is deterministic — the server checks file extensions in the diff, not an LLM.

### Relationship to Adversarial Review

Playwright results feed INTO the adversarial review, not replace it:
- Gemini (Challenger) sees: "Playwright: 3 passed, 1 failed — test_project_selector_visible FAILED: Timeout waiting for #project-selector"
- This is concrete evidence for the Challenger to cite: "The mission is NOT achieved — the project selector doesn't render"
- Claude (Author) can't rebut a failing functional test with hand-waving

### What This Does NOT Cover

- Visual/screenshot verification ("does it look right" — separate spec for later, LLM judges screenshot)
- Performance testing (load, latency)
- Mobile/responsive testing
- Tests for non-web projects (CLI, API-only — different verification strategy needed)

## Prerequisites
- spec:014 (Autonomous Runtime) — the pipeline that runs builds
- spec:011 (Adversarial Evaluation) — the review pipeline that receives test results

## Key Decisions
- Gemini writes the test, not Claude. The builder should not write its own acceptance test — that's the adversary's job.
- Tests are pytest-based (pytest-playwright), not a separate framework. One `run_tests()` call runs everything.
- The server decides when to run Playwright (based on file extensions in diff), not an LLM.
- Functional tests are generated per-spec, not per-Done-When-item. One test file per spec.
